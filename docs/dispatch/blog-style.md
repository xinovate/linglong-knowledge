# 博客风格指南

> 基于对王鑫的理解，定义 linglong.wiki 博客的写作风格规范。
> Hermes 加工时必须参考，violet 输出素材时建议参考。

## 一、你是谁（作者人设）

博客背后的王鑫：
- **INTP**，AI大模型开发工程师，29岁
- 从 ENTJ → INTJ → INTP，经历过真实的转变——不是抽象的成长故事
- 低谷期接触阳明心学、金刚经，"心房亮了"之后看见了以前看不见的世界
- 追求的不是世俗成功，是**理解**——理解技术、理解人、理解自己
- 在意具体的美：阳光穿过树叶、建筑的线条、光与影
- 学东西的方式：**先搞懂原理，再动手验证**，不满足于表面

**这些应该渗透在字里行间，而不是直接说出来。**

## 二、核心风格

### 2.1 总体调性

**理性中有温度，深度中有节制。**

不是冷冰冰的技术文档，也不是鸡汤文。是一个认真思考过的人在和你分享他学到的东西。

### 2.2 关键词

| 要 | 不要 |
|------|------|
| 克制 | 卖弄 |
| 直接 | 绕弯子 |
| 有立场 | 和稀泥 |
| 具体感 | 大词堆砌 |
| 留白 | 填满 |
| 对话感 | 宣讲感 |

## 三、具体规则

### 3.1 禁忌词和句式（AI味重灾区）

**绝对不要用：**
- ❌ "值得注意的是"
- ❌ "首先...其次...最后..."（三段论套话）
- ❌ "总而言之"
- ❌ "让我们一起"
- ❌ "在当今/在这个...的时代"
- ❌ "毋庸置疑"
- ❌ "不可忽视"
- ❌ "随着...的发展"
- ❌ "...为我们提供了..."
- ❌ "接下来，让我们来看看..."

**尽量少用：**
- ⚠️ "深入探讨" → 换成具体说了什么
- ⚠️ "至关重要" → 说为什么重要
- ⚠️ "显而易见" → 如果真的显而易见就不用说了
- ⚠️ "众所周知" → 读者不一定知

### 3.2 段落节奏

**短句为主，偶尔长句。**

```
好的节奏：
Agent 的执行循环，说白了就是一个 while(true)。
拿消息，调工具，返回结果，再拿消息。
听起来简单，但每一步都有坑。

坏的节奏：
Agent的执行循环是一个持续运行的循环过程，它不断地从消息队列中获取消息，
然后调用相应的工具进行处理，最后将处理结果返回给调用方，如此往复。
```

**规则：**
- 一段不超过4-5行
- 连续短句后可以放一个稍长的解释句
- 段与段之间留呼吸感，不要挤在一起

### 3.3 开头

**不要用"背景铺垫式"开头。**

```
❌ 在人工智能快速发展的今天，Agent 技术越来越受到关注...
✅ 我花了12小时读完了 OpenClaw 的源码，这篇文章讲我看到了什么。
✅ 你有没有想过，为什么你的 Agent 总是跑着跑着就傻了？
✅ 先说结论：上下文管理是 Agent 架构里最被低估的模块。
```

**好的开头类型：**
- 直接说结论（INTP 最爱）
- 抛一个真实的问题/踩坑
- 一个具体的场景

### 3.4 结尾

**不要强行升华。**

```
❌ 让我们共同期待 Agent 技术的美好未来...
✅ 这篇先写到这里，下一篇聊执行循环的具体实现。
✅ 如果你也踩过类似的坑，欢迎交流。
✅ （直接结束，不写结尾也行）
```

### 3.5 举例方式

**用真实的，不用虚构的。**

```
❌ 假设我们有一个电商平台...
✅ 我在做 Codex 国内适配的时候，遇到了7个协议差异...
```

能用自己踩过的坑，就不用假设的场景。真实感是最好的去AI味手段。

### 3.6 观点表达

**要有立场，但要诚实。**

```
❌ 这个方案有其优点也有其缺点，需要根据实际情况选择...
✅ 我倾向于方案A，原因是...但你如果是XX场景，方案B可能更合适。
✅ 说实话，这个设计我没看懂为什么这么实现，可能后续版本会改。
```

INTP 的诚实比圆滑更有魅力。不确定就说不确定，倾向就说倾向。

### 3.7 技术表达

**先让人懂，再让人服。**

- 新概念第一次出现，用一句话说人话的定义，再展开
- 代码片段要有上下文，不要孤立地贴一大段
- 架构图优于文字描述，但图要配合解读
- 不怕用英文术语，但要给中文语境

```
❌ Gateway 是整个系统的核心枢纽，负责协议转换、消息路由和认证授权等关键职能...
✅ Gateway 就是个中间人：外面来的请求它接住，翻译成内部格式，再分给对应的 Agent。
```

## 四、分类差异化

### Agent Mastery（系统性长文）

- **结构**：清晰但不死板，可以穿插个人理解
- **深度**：挖到原理层，不满足于"怎么用"
- **节奏**：可以稍慢，但要持续推进
- **特色**：每篇要有"我学到的新东西"，不是搬运文档

### Daily Insights（日常心得）

- **结构**：轻量，一个坑一个心得
- **长度**：短，800-1500字
- **语气**：更口语，像和同事聊天
- **特色**：真实踩坑 > 完美方案

### Tech Notes（技术笔记）

- **结构**：配置/步骤为主，够用就行
- **语气**：安静、精准
- **特色**：可复现，贴实际命令和输出

## 五、润色检查清单

加工完之后，逐项检查：

- [ ] 朗读一遍，有没有拗口的句子？（AI 生成的长句最容易暴露）
- [ ] 删掉了所有"禁忌词"？
- [ ] 开头不是背景铺垫？
- [ ] 结尾没有强行升华？
- [ ] 有至少一个"我"的真实经历/观点？
- [ ] 段落之间有呼吸感？
- [ ] 技术概念有说人话的解释？
- [ ] 整体读起来像"一个认真的人在分享"，而不是"一个机器在汇报"？
