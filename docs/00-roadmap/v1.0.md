# v1.0 路线图：跨 Agent 知识中枢

## 愿景

Linglong v1.0 是一个**跨 Agent 知识中枢**，解决的核心问题是：

> OpenClaw、Claude Code、Codex 等 AI Agent 各自维护独立的知识库，互不相通。同一个概念，OpenClaw 知道，Claude Code 不知道；Claude Code 记了，Codex 又记一遍。

**v1.0 目标**：建立"信息获取 → 知识沉淀 → 内容编译 → 智能分发"的完整闭环，让任意 Agent 的产出都能被统一管理和复用。

## 详细规划

### v0.4：知识库统一（最高优）

**核心问题**：OpenClaw 的 wiki 在 `~/.openclaw/workspace/memory/wiki/`，Claude Code 的 memory 在 `~/.claude/projects/.../memory/`，两者互不相通。

**交付物**：

1. **兼容 OpenClaw wiki 结构**
   - `concepts/`、`experiences/`、`projects/`、`references/`、`methodologies/`、`user/`、`emotion/`
   - 支持 `[[wikilinks]]` 格式
   - 文件系统布局与 OpenClaw 保持一致

2. **向量搜索落地**
   - 复用 OpenClaw 远程 embedding 服务（`http://localhost:7997`，nomic-embed-text-v1.5）
   - 本地 fallback：sqlite-vec
   - 混合搜索（hybrid search）+ MMR + 时间衰减

3. **跨 Agent 同步协议**
   - 定义 Agent 命名空间：`openclaw:`、`claude:`、`codex:`
   - 同步方向：双向（Linglong 作为 Source of Truth，各 Agent 定期 sync）
   - 冲突解决：时间戳优先 + 人工确认

4. **短期→长期转换**
   - 继承 OpenClaw MEMORY.md 规则：任务完成后判断是否有长期价值
   - 自动从 `memory/YYYY-MM-DD/` 迁移到 `memory/wiki/`

**验收标准**：
- OpenClaw 新写入的 wiki 条目能在 Linglong 中搜索到
- Claude Code 的 memory 文件能同步到 Linglong knowledge store
- 向量搜索能返回语义相关的结果

---

### v0.5：ingest 通用化

**核心问题**：ai-morning-brief skill 已经跑通了多源采集 + 真实性验证 + 定时调度，但它是硬编码的（只抓 AI 领域、只输出早报格式）。

**交付物**：

1. **通用信息源适配器**
   - Web Search（DuckDuckGo / 必应 CN）
   - Web Fetch（并行抓取，支持代理/无代理双模式）
   - API 调用（GitHub API、Twitter API、自定义 REST）
   - RSS（已有基础）
   - 爬虫（Playwright / Scrapy 预留接口）

2. **可配置"信息源套餐"**
   - YAML 定义采集策略
   - 支持任意主题（不只是 AI）
   - 支持 cron 定时调度 + 手动触发 + 事件驱动

3. **真实性验证引擎**
   - 继承 ai-morning-brief 的 5 层检查机制
   - 多源交叉验证、数字合理性、时间有效性、源头权威、常识判断

**验收标准**：
- 用 YAML 配置一个"产品动态监控"源，能自动采集并验证
- 真实性验证能过滤虚假数据（如异常融资金额）

---

### v0.6：多 Agent 接入

**核心问题**：Claude Code 和 Codex 的记忆系统独立于 OpenClaw，没有统一入口。

**交付物**：

1. **Claude Code 同步适配器**
   - 读取 `~/.claude/projects/.../memory/` 下的 `user/`、`feedback/`、`project/`、`reference/`
   - 映射到 Linglong 的 `/wiki/user/`、`/wiki/experiences/`、`/wiki/projects/`、`/wiki/references/`

2. **Codex 同步适配器（预留）**
   - 调研 Codex 的记忆存储位置
   - 设计同步接口

3. **写入冲突解决**
   - 多个 Agent 同时修改同一篇 wiki → 有 merge 策略
   - 时间戳优先，冲突时标记为 `pending_review`

**验收标准**：
- Claude Code 的 memory 修改后能自动同步到 Linglong
- 两个 Agent 同时修改同一文件时，不会丢失数据

---

### v0.7：composer 产品化

**核心问题**：composer 只能输出博客，形式单一。

**交付物**：

1. **多模板支持**
   - `blog`：Hexo 博客（已有）
   - `morning-brief`：AI 早报（继承 ai-morning-brief 输出格式）
   - `weekly`：周报
   - `ppt`：PPT 大纲（用于抖音口播脚本）
   - `twitter-thread`：Twitter 线程

2. **AI 封面图生成**
   - 接入 DALL-E / Stable Diffusion / 图虫
   - LLM 根据文章内容生成图像提示词
   - 异步生成，不阻塞流水线

3. **内容验证**
   - 检查是否符合模板规范（标题长度、摘要长度、标签格式等）
   - frontmatter 复杂 YAML 支持（list 格式）

**验收标准**：
- 同一批知识碎片能生成博客、早报、PPT 三种不同格式的输出
- AI 封面图生成失败时自动回退到占位图

---

### v0.8：dispatch 正式化

**核心问题**：内容只能发博客，缺少多平台分发能力。

**交付物**：

1. **发布队列**
   - 状态：待发布 / 发布中 / 已发布 / 失败
   - 失败可重试
   - 定时发布支持

2. **多平台路由**
   - 博客 → Hexo → Git Workflow Publisher（已有）
   - 早报 → 钉钉（复用 OpenClaw dingtalk-connector）
   - 短视频脚本 → 抖音（脚本 + 封面图，预留）
   - Twitter 线程 → Twitter API（预留）

3. **反馈回流**
   - 阅读量、点赞、评论收集
   - 回流到 knowledge，更新实体状态

**验收标准**：
- 一篇博客文章能自动发布到 Hexo 并触发服务器生成
- 发布失败时能自动重试并记录错误

### v1.0：发布

**验收标准**：

1. **任意 Agent 产出 → 统一知识库**
   - OpenClaw、Claude Code、Codex 的新产出能在 1 分钟内同步到 Linglong

2. **统一知识库 → 智能编译**
   - 能从知识库中自动聚类、提炼、生成多格式内容
   - 内容质量通过模板验证

3. **智能编译 → 多平台分发**
   - 博客、早报、短视频脚本能自动分发到对应平台
   - 发布成功率和失败重试机制可靠

4. **完整闭环**
   - 从信息获取到平台分发，全程无需手动改代码
   - 人工只在审核环节介入

## 关键架构决策

### ADR-001：Linglong 作为跨 Agent 知识中枢

**决策**：Linglong 是所有 AI Agent 的统一知识底座，各 Agent 通过它读写知识，而不是各自维护独立知识库。

**理由**：
- 避免重复存储同一知识
- 保证多个 Agent 之间的知识一致性
- 便于知识复用和交叉引用

**状态**：已接受

### ADR-002：知识库同步方向

**决策**：Linglong 作为 Source of Truth，各 Agent 定期双向同步。

**理由**：
- 单一真相源避免冲突
- 双向同步保证各 Agent 能及时获取最新知识
- 冲突时以 Linglong 为准，标记待审

**状态**：已接受

### ADR-003：向量搜索双模式

**决策**：优先使用 OpenClaw 远程 embedding 服务，本地 sqlite-vec 作为 fallback。

**理由**：
- OpenClaw 已有成熟的 embedding 服务（nomic-embed-text-v1.5，混合搜索 + MMR）
- 本地 fallback 保证离线可用
- 避免重复部署 embedding 服务

**状态**：已接受
