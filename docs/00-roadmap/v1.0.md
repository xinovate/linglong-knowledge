# v1.0 路线图：博客流水线

## 愿景

Linglong v1.0 是一个**可用的博客流水线**——多 Agent 知识统一写入 KnowledgeStore，Composer 按规则生成博客文章并配图，Dispatch 发布到 Hexo 博客系统。

**v1.0 目标**：跑通博客闭环。多 Agent 正常写入读取、Composer 按规则生成博客、配图系统从 URL 列表获取背景图和文章配图、Dispatch 能发到博客系统。

**v2.0 目标**：产品化——多模板（早报/周报/PPT）、AI 封面图、WebSearchAdapter、发布队列与重试、API 冻结。

## 详细规划

### v0.4：知识库统一（最高优）

**状态**：✅ 已完成（OpenClaw + Claude Code + Codex 已接入，向量搜索已落地）

**核心问题**：OpenClaw 的 wiki 在 `~/.openclaw/workspace/memory/wiki/`，Claude Code 的 memory 在 `~/.claude/projects/.../memory/`，两者互不相通。

**交付物**：

1. **兼容 OpenClaw wiki 结构** ✅
   - `concepts/`、`experiences/`、`projects/`、`references/`、`methodologies/`、`user/`、`emotion/`
   - 支持 `[[wikilinks]]` 格式
   - `OpenClawSyncAdapter` 已实现（`ccd2011`）

2. **Claude Code memory 同步** ✅
   - 读取 `~/.claude/projects/.../memory/` 下 `.md` 文件
   - 映射类型到 wiki 目录（`feedback`→`experiences/` 等）
   - `ClaudeCodeSyncAdapter` 已实现（`ffb7c10`），含冲突检测

3. **向量搜索落地** ✅
   - 复用 OpenClaw 远程 embedding 服务（`http://localhost:7997`，nomic-embed-text-v1.5）
   - `EmbeddingGenerator` 支持 OpenAI-compatible API，失败时返回 None 不阻塞存储
   - `KnowledgeStore.search_similar()` 基于 `sqlite-vec` `vec_distance_cosine`
   - `sqlean.py` fallback 解决 macOS 等系统 `sqlite3` 缺少 `load_extension` 的问题
   - 混合搜索（hybrid search）+ MMR + 时间衰减 预留到 v0.7

4. **跨 Agent 同步协议** ✅（基础版）
   - ADR-004：Agent 命名空间前缀（`openclaw:`、`claude:`、`codex:`）
   - ADR-005：Memory 类型映射（Claude Code `type` → wiki 目录）
   - 冲突解决：跳过已存在实体（`skipped` 计数）

5. **短期→长期转换** ⏳
   - 继承 OpenClaw MEMORY.md 规则：任务完成后判断是否有长期价值
   - 自动从 `memory/YYYY-MM-DD/` 迁移到 `memory/wiki/`

**验收标准**：
- ✅ OpenClaw 新写入的 wiki 条目能在 Linglong 中搜索到
- ✅ Claude Code 的 memory 文件能同步到 Linglong knowledge store
- ✅ 向量搜索能返回语义相关的结果

**交付物**：

2. **通用信息源适配器** ✅

5. **PackageExecutor 并行执行** ✅
   - 并发抓取包内所有启用源
   - 去重、验证、审核、存储完整流水线
   - 错误隔离：单个源失败不影响其他源

**待增强（v0.5+）**：

- Playwright / Scrapy 爬虫预留接口

**验收标准**：
- ✅ 用 YAML 配置一个采集包，PackageExecutor 能自动采集并验证
- ✅ 真实性验证能过滤虚假数据（如异常融资金额、过期内容）

## 关键架构决策

### ADR-001：Linglong 作为跨 Agent 知识中枢

**决策**：Linglong 是所有 AI Agent 的统一知识底座，各 Agent 通过它读写知识，而不是各自维护独立知识库。

**理由**：
- 避免重复存储同一知识
- 保证多个 Agent 之间的知识一致性
- 便于知识复用和交叉引用

**状态**：已接受

### ADR-002：知识库同步方向

**决策**：Linglong 作为 Source of Truth，各 Agent 定期双向同步。

**理由**：
- 单一真相源避免冲突
- 双向同步保证各 Agent 能及时获取最新知识
- 冲突时以 Linglong 为准，标记待审

**状态**：已接受

### ADR-003：向量搜索双模式

**决策**：优先使用 OpenClaw 远程 embedding 服务，本地 sqlite-vec 作为 fallback。

**理由**：
- OpenClaw 已有成熟的 embedding 服务（nomic-embed-text-v1.5，混合搜索 + MMR）
- 本地 fallback 保证离线可用
- 避免重复部署 embedding 服务

**状态**：已接受

### ADR-004：Agent 命名空间前缀

**决策**：各 Agent 的实体 ID 必须带命名空间前缀，避免同一概念在不同 Agent 中冲突。

**理由**：
- OpenClaw 和 Claude Code 可能有同名文件（如 `user/profile.md`）
- 不带前缀会导致后同步的覆盖先同步的
- 前缀格式：`{agent}:{sha256(path)[:16]}`

**实现**：
- OpenClaw: `openclaw:{sha256(relative_path)[:16]}`
- Claude Code: `claude:{sha256(filename)[:16]}`
- Codex（预留）: `codex:{sha256(filename)[:16]}`

**状态**：已接受（2026-05-12）

### ADR-005：Memory 类型映射

**决策**：Claude Code memory 的 `type` 字段映射到 Linglong wiki 目录结构。

**映射表**：

| Claude Code type | Linglong wiki 目录 | 理由 |
|------------------|-------------------|------|
| `feedback` | `experiences/` | 反馈属于经验 |
| `project` | `projects/` | 项目状态 |
| `user` | `user/` | 用户画像 |
| `reference` | `references/` | 参考资料 |
|（无 type 或未知）| `concepts/` | 默认归类 |

**状态**：已接受（2026-05-12）
