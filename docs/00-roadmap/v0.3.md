# v0.3 人工审核层

v0.3 的目标是在"全自动发布"与"完全手动"之间建立可控的中间态：流水线生成草稿后，由人工确认内容质量、标题准确性和封面图适配度，再执行正式发布。同时补齐 Git 工作流发布器和 AI 封面图生成，使生产环境部署更加可靠。

---

## 1. Draft Mode 草稿模式

**描述**：流水线生成文章后，不直接发布，而是进入草稿状态等待人工审核。审核通过后再触发正式 publish。

**实现要点**：
- 新增 `Composer.run(draft=True)` 模式：
  - 完整执行提取、聚合、提炼、模板渲染流程
  - 生成的 Markdown 文件写入独立的草稿目录（由 `ComposerConfig.drafts_dir` 配置）
  - 同时生成一份审核摘要（`review.md`），包含：标题、摘要、标签、涉及的记忆片段来源、字数统计
- DraftManager API：
  - `save_draft()`：保存草稿到独立目录
  - `list_drafts(status=...)`：按状态列出草稿
  - `get_draft(draft_id)`：获取草稿详情
  - `publish_draft(draft_id)`：发布草稿，返回 dispatch-ready payload
  - `discard_draft(draft_id, keep=False)`：废弃草稿
- 草稿状态持久化到文件系统，记录：草稿 ID、生成时间、关联的片段哈希、审核状态
- 审核通过后，草稿标记为 `published`，关联的片段哈希写入 `ComposerState`，避免重复生成

**验收标准**：
- `Composer.run(draft=True)` 生成文章但不直接发布
- `DraftManager.list_drafts()` 列出所有待审核草稿，包含标题和生成时间
- `DraftManager.get_draft()` 可查看完整 Markdown 内容
- `DraftManager.publish_draft()` 将草稿标记为发布状态并返回 dispatch-ready 结果
- 已发布草稿关联的片段哈希被记录，避免重复生成
- 废弃草稿可选择是否保留为本地文件

**状态**：✅ 已完成

## 2. Git Workflow Publisher

**描述**：替代直接调用 `hexo generate` 的发布方式，改为"生成文件 → Git commit → Git push → 服务器端生成"的工作流，适配生产环境部署。

**实现要点**：
- Publisher 文件已迁移到 `src/linglong/dispatch/publishers/`（v0.8 正式化）。
- 支持 git add → commit → push → SSH 远程触发完整链路。
- 含代理配置和错误回退。

**验收标准**：
- 文章生成后自动 commit 并 push 到远程仓库
- 提交信息格式规范，包含文章标题和日期
- SSH 远程触发成功时，服务器端正确生成静态页面
- Git 操作失败时返回明确的错误信息，不破坏工作区状态

**状态**：✅ 已完成（代码已迁移到 dispatch 占位目录）

## 3. AI 封面图生成

**描述**：利用 AI 图像生成模型（如 DALL-E、Midjourney、Stable Diffusion 等）根据文章标题和内容自动生成高质量封面图，替代当前的纯色占位图。

**实现要点**：
- 在 `ImageAssetGenerator` 中实现 `_generate_ai_cover()` 方法（当前为预留空实现）
- 封面图策略优先级中 `ai_generate` 从预留变为实际可用
- 需要 LLM 先根据文章内容生成图像提示词（image prompt），再调用图像生成 API
- 支持多种 AI 图像提供商（通过配置切换）

**配置项（`core/config.py` 中的 `ComposerConfig`）**：

```python
class ComposerConfig(BaseSettings):
    # ...
    assets_cover_enabled: bool = False
    # 未来扩展：
    # assets_cover_provider: str = "openai"
    # assets_cover_model: str = "dall-e-3"
```

**验收标准**：
- `assets_cover_enabled=True` 且启用时，自动生成与文章主题相关的封面图
- 图像生成失败时自动回退到下一优先级策略
- 生成的图片尺寸和格式符合博客主题要求
- 支持通过配置切换不同的 AI 图像提供商
- 图像生成耗时较长时，不阻塞流水线其他步骤（可考虑异步或后台生成）

**状态**：🟡 待讨论（依赖外部 API，需考虑成本和超时）

## 4. 审核与发布的联动设计

**描述**：草稿模式、Git 工作流和 AI 封面图三个功能在 v0.3 中需要协同工作，形成完整的人工审核闭环。

**预期工作流**：

```
1. 记忆源产生新内容
   ↓
2. Composer.run(draft=True)
   ↓
3. 流水线生成草稿 + AI 封面图（异步）
   ↓
4. 人工审核：查看草稿详情
   - 检查标题是否准确（10–18 汉字）
   - 检查摘要质量（30–40 汉字）
   - 检查封面图是否合适
   - 如需修改：直接编辑草稿 Markdown 文件
   ↓
5. 确认发布：DraftManager.publish_draft()
   - 返回 dispatch-ready payload
   - 由 dispatch 模块执行 Git add / commit / push
   - 可选：SSH 触发服务器端生成
   ↓
6. 发布完成，状态更新
```

**验收标准**：
- 整个审核闭环可在 API/CLI 完成，无需手动操作 Git 或复制文件
- 审核过程中修改的 Markdown 内容在正式发布时保留
- 同一批记忆片段在草稿废弃后，可重新生成新草稿（不违反去重规则）

**状态**：🟡 待讨论
